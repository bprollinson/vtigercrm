vtiger CRM 5 Alpha - Security Management Implementation

Date: January 16, 2006

The following functions are incorporated in the vtiger CRM 5 Alpha Security Management implementation:

1. The Role is based on Organization Hierarchy.

2. The Roles can have one-to-multiple profiles.

3. Each User is associated with a role, which in turn associated with the profiles.

4. Profiles specify the access for various actions in each of the modules.

5. Group Support- Group can consists of:

  • Groups
  • Roles
  • Roles & Subordinates
  • Users

Note: In vtiger CRM 5.0 Alpha group level security has been implemented only for the Leads, Tickets, FAQ, and Activities modules.

6.  Profiles override the Sharing Rules i.e. the Profile is the master:

For example, one of the users has sharing privilege to view others accounts. But his profile does not allow him to view accounts. In this case,  the user cannot view the accounts.

7. Default Organization Sharing Access:

Your organization's sharing model sets the default access that users have to each other's data. The default sharing access levels are listed below. 

Type Default Access Level Controlled By
Accounts Public Read/Write Organization-wide defaults
Activities Private

Access to other users' activities is based on your  position in the hierarchy.

Note that access to events in a calendar view is controlled by the organization-wide calendar sharing.
Calendar Access Hide Details and Add Events Organization-wide defaults
Tickets Public Read/Write Organization-wide defaults
Contacts Public Read/Write Organization-wide defaults
Emails Public Read/Write Organization-wide defaults
Leads Public Read/Write Organization-wide defaults
Opportunities Public Read Only Organization-wide defaults
Quotes Public Read/Write Organization-wide defaults 
Sales Order Public Read/Write Organization-wide defaults
Purchase Order Public Read/Write Organization-wide defaults
Invoice Public Read/Write Organization-wide defaults 

The following Sharing Permissions can be set :

Field Description
Private Only the record owner, and users above that role in the hierarchy, can view, edit, and report on those records.
Public Read Only All users can view and report on records but not edit them. Only the owner, and users above that role in the hierarchy, can edit those records.
Public Read/Write All users can view, edit all records.
Public Read/Write/Delete All users can view, edit and delete all records

Note: When Account is Private, the Related Potential, Ticket, Quote, Sales Order, Purchase Order, and Invoice access is also set to private in Default Organization Sharing model.

8.  Sharing Rules Specification by Administrator:

  • Apart from the Default Organization Sharing Access, the administrator  can also create Organization Level Sharing Rules for the following modules to share the data between Roles, Roles&Subordinates and Groups :
    • Leads
    • Accounts/Contacts
    • Potentials
    • Emails
    • HelpDesk
    • Quotes
    • SalesOrder
    • Purchase Order
    • Invoices
  • Sharing Rules can only extend the visibility and cannot hide the visibility     
  • On creating the Sharing Rules, Access for the Related modules can also be specified.
  • Sharing Rules are created to share data between Roles, Roles&Subordinates and Groups

Notes on Data Sharing:

  • Regardless of the organization-wide defaults, users can always view and edit all data owned by or shared with users below them in the role hierarchy.
  • While your sharing model controls visibility to records, user permissions control what users can do to the records that are visible to them. For example, if you share an account with other users, those users can only see the account if they have the "Read" permission on accounts. Likewise, users who have the "Edit" permission on contacts may still not be able to edit contacts they do not own if they are working in a Private sharing model.
  • Administrators, and users with the "View All Data" or "Edit All Data" permissions, have access to view or edit all data.
  • Only Administrators have access to configure the Access Privileges   
  • You must have at least read access to a record to be able to add activities or other associated records to it (for example associated products)

IMPORTANT:

The new security management support is provided for the following:

  • Home Page, Custom View, Reports, Dashboard, Activity History, and Calendar.
  • Delete Handling of Roles, Groups is not fully functional.
  • Editing of Groups will not function properly.
  • In ListView, Related List View, Popup List View & Search List View Sharing Rules for Related Modules are not implemented.
  • Tab hiding in header is not implemented.
  • Group Level Security support is implemented only for the Leads, Activities and Tickets modules.
  • Parent Type Security Handling is not implemented for the Products and Notes modules.
  • Read Only Support in field level security is not implemented.
